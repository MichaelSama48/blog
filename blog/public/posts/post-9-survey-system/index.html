Chủ đề 9: Dự án thực tế - Hệ thống Bỏ phiếu Trực tuyến & Bài toán Concurrency

Không chỉ là CRUD. Làm sao để đếm chính xác 1 triệu lượt vote trong 1 giây? Giải quyết Race Condition và xây dựng Dashboard Real-time với Spring Boot & Redis.
June 25, 2024 · 4 min

Nếu ứng dụng Chat (ở bài trước) thiên về trải nghiệm người dùng, thì Hệ thống Khảo sát/Bỏ phiếu (Voting System) là bài toán kinh điển để kiểm tra trình độ thiết kế hệ thống (System Design) của bạn, đặc biệt là với sinh viên ngành Hệ thống Thông tin.

Tại sao? Vì nó đụng đến vấn đề nhức nhối nhất của lập trình mạng: Xử lý đồng thời (Concurrency) và Toàn vẹn dữ liệu (Data Integrity). Hãy tưởng tượng đêm chung kết Rap Việt, MC Trấn Thành hô “Bắt đầu bình chọn”. 1 triệu người cùng bấm nút trong 1 giây. Nếu bạn code không khéo, Database sẽ sập, hoặc tệ hơn: Người dùng vote rồi nhưng hệ thống không đếm.

Bài viết này sẽ hướng dẫn bạn xây dựng backend cho một hệ thống như thế.

1. Bài toán Race Condition (Điều kiện đua)#

Giả sử trong Database có bảng Poll:

  • option_a: 100 phiếu.

Khi User A và User B cùng vote cho A cùng lúc:

  1. Thread A đọc Database: count = 100.
  2. Thread B đọc Database: count = 100.
  3. Thread A tăng 1 -> 101 -> Ghi xuống DB.
  4. Thread B tăng 1 -> 101 -> Ghi xuống DB.

Kết quả: Mất 1 phiếu! Đây là Race Condition.

2. Giải pháp Kiến trúc: CQRS + Redis#

Để chịu tải cao, chúng ta không cho User ghi thẳng vào Database. Chúng ta dùng kiến trúc đệm.

  1. Write Side (Ghi): User vote -> REST API -> Redis Queue. (Phản hồi ngay lập tức “Đã ghi nhận”, cực nhanh).
  2. Worker (Xử lý): Một process chạy ngầm (Background Worker) lấy từng phiếu từ Queue ra và cộng dồn vào Database một cách tuần tự (hoặc dùng Atomic Increment).
  3. Read Side (Đọc): Dashboard kết nối WebSocket để nhận kết quả Real-time từ Redis Pub/Sub.

3. Tech Stack#

  • Backend: Java Spring Boot (Mạnh mẽ, xử lý Transaction tốt).
  • Cache/Queue: Redis (Tốc độ ánh sáng).
  • Database: MySQL (Lưu trữ bền vững).
  • Frontend: JavaScript thuần (hoặc React).

4. Triển khai Backend (Spring Boot)#

4.1. Cấu hình Redis Atomic#

Thay vì cộng trừ trong Java (dễ lỗi thread), ta dùng lệnh INCR của Redis. Đây là lệnh nguyên tử (Atomic), đảm bảo 100% không bao giờ sai số dù 1 tỷ người gọi cùng lúc.

@Service
public class VotingService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Autowired
    private SimpMessagingTemplate messagingTemplate; // WebSocket

    public void castVote(String pollId, String optionId) {
        // 1. Tăng biến đếm trong Redis (Atomic Increment)
        // Key: "poll:123:option:A"
        String key = "poll:" + pollId + ":option:" + optionId;
        Long newCount = redisTemplate.opsForValue().increment(key);

        // 2. Bắn sự kiện Real-time qua WebSocket cho Dashboard
        VoteUpdate update = new VoteUpdate(pollId, optionId, newCount);
        messagingTemplate.convertAndSend("/topic/poll/" + pollId, update);
        
        // 3. (Optional) Async Worker sẽ sync từ Redis về MySQL sau mỗi 1 phút
    }
}

4.2. WebSocket Controller (Gửi kết quả)#

@Controller
public class PollController {

    @Autowired
    private VotingService votingService;

    // API nhận Vote qua HTTP POST (cho đơn giản)
    @PostMapping("/api/polls/{id}/vote")
    @ResponseBody
    public ResponseEntity<?> vote(@PathVariable String id, @RequestBody VoteRequest request) {
        votingService.castVote(id, request.getOptionId());
        return ResponseEntity.ok("Vote success");
    }
    
    // WebSocket endpoint để Dashboard lắng nghe
    // Client subscribe: /topic/poll/123
}

5. Triển khai Frontend (Dashboard)#

Dashboard sẽ hiển thị biểu đồ thanh (Bar Chart) chạy realtime.

// Kết nối WebSocket (dùng thư viện SockJS và STOMP của Spring)
const socket = new SockJS('/ws');
const stompClient = Stomp.over(socket);

stompClient.connect({}, function (frame) {
    console.log('Connected: ' + frame);
    
    // Đăng ký nhận tin của Poll ID 123
    stompClient.subscribe('/topic/poll/123', function (message) {
        const update = JSON.parse(message.body);
        updateChart(update.optionId, update.newCount);
    });
});

function vote(option) {
    // Gọi REST API để vote
    fetch('/api/polls/123/vote', {
        method: 'POST',
        headers: {'Content-Type': 'application/json'},
        body: JSON.stringify({ optionId: option })
    });
}

6. Bài toán mở rộng (Scaling)#

Nếu Rap Việt có 10 triệu người xem thì sao?

  1. Load Balancer: Dựng 10 con Server Spring Boot đằng sau Nginx.
  2. Redis Cluster: 1 con Redis không đủ chứa, phải dùng Cluster sharding.
  3. Kafka: Thay Redis Queue bằng Apache Kafka để đảm bảo không mất phiếu nếu sập điện.

7. Kết luận#

Dự án Hệ thống Bỏ phiếu này giúp bạn học được:

  1. Tránh Locking DB: Đừng bao giờ UPDATE polls SET count = count + 1 khi tải cao. Chết Database đấy!
  2. Sức mạnh của Redis: Atomic Counter là cứu cánh cho bài toán đếm.
  3. Mô hình Event-Driven: User Vote -> Event -> WebSocket Client.

Đây là một đồ án “điểm 10” cho môn Lập trình mạng hoặc Hệ thống phân tán, chứng tỏ bạn hiểu sâu về cách dữ liệu di chuyển và được bảo vệ trong môi trường khốc liệt.
